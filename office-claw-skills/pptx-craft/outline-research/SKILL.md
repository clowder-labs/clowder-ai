# 大纲驱动的两阶段研究技能（Outline-Driven Research）

## 角色定位

你是一位**结构化研究策划师**：先调研摸底，再生成有事实支撑的大纲，最后定向深研。与 planner（基于现有认知直接规划）不同，本技能适用于用户主题宽泛、需要外部事实和数据支撑的场景。

## 核心原则

1. **调研先行**：不凭空编大纲，先 3-5 次搜索摸清领域全貌，大纲每页主题有真实信息支撑
2. **研究分两段**：广度检索铺开来源 → 深度抓取提取可上页信息
3. **证据驱动**：每页成果至少含一类硬信息（具体数据 / 明确案例 / 可验证事实）
4. **自主执行**：仅在需求不可理解、矛盾要求、搜索严重不足（<3 有效来源）时暂停

## 输入输出

### 输入

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `topic` | string | — | **必需**，用户的研究主题 |
| `output_base` | string | 本技能目录下 `output/` | 输出基础目录（自动创建时间戳子目录） |
| `output_dir` | string | 自动生成 | 用户显式指定完整路径时使用，跳过时间戳目录 |
| `page_count` | number | 10 | 目标页数（3-30） |
| `audience` | string | "通用商务/知识分享" | 目标受众 |
| `focus_areas` | string | 自动推断 | 研究重点方向 |
| `research_depth` | string | "L2" | L1(≥3k字,≥10源,2-4轮) / L2(≥5k字,≥20源,5-8轮) / L3(≥8k字,≥30源,8-12轮) |

### 输出

| 文件 | 说明 |
|------|------|
| `{output_dir}/outline.md` | PPT 大纲（Markdown，含研究标记） |
| `{output_dir}/research.md` | 按页映射的深度研究报告（含结构化数据表格和可视化建议） |

## 工作流程概览

```
用户需求
  → Stage 1: 需求调研
  → Stage 2: 大纲生成
  → Stage 3: 广度检索
  → Stage 4: 反思与迭代
  → Stage 5: 深度抓取
  → Stage 6: 成果撰写
  → Stage 7: 验证与交付
```

## Stage 1: 需求调研

### 1.1 初始化输出目录

| 条件 | 行为 |
|------|------|
| 用户未指定路径 | `node {skill_root}/pptx-craft/scripts/utils/generate_timestamp_dir.js [output_base]`，返回路径赋值给 `{output_dir}` |
| 用户指定完整路径 | 直接使用，`mkdir -p {output_dir}` |

### 1.2 需求标准化

从用户输入提取：主题(`topic`)、受众(`audience`，默认"通用商务/知识分享")、页数(`page_count`，默认10)、研究深度(`research_depth`，默认L2)。

### 1.3 快速调研

发起 **5-8 次并行搜索**，快速掌握：
1. 领域现状（整体格局）
2. 关键维度（业内讨论的核心方面）
3. 最新动态（近期重要事件/数据/趋势）
4. 核心玩家（关键机构/公司/人物）
5. 争议热点（分歧观点或热门话题）

搜索查询覆盖：综述概览、英文最新趋势、核心技术/挑战、市场数据、企业案例。添加年份限定（如"2026"、"latest"）和可信来源关键词（如"report"、"statistics"、"白皮书"）。中文主题用中英双语查询。

### 1.4 调研整理

将搜索结果整理为结构化摘要（存入上下文，不写文件）：领域概况、关键维度、重要发现（附来源）、关键数据点（附来源）、核心玩家/案例线索、大纲建议方向。

## Stage 2: 大纲生成

基于用户需求 + Stage 1 调研结果，生成结构化大纲，写入 `{output_dir}/outline.md`。

### 生成规则

1. 大纲基于调研中的真实发现，不编造不存在的趋势或数据
2. 每页一行：页码 | 类型 | 标题 | 研究需求 | 研究查询 | 数据需求 | 视觉策略
3. 类型为 intro/agenda/conclusion 的页面研究需求标记 ❌，trend/data/case/comparison/technology 标记 ✅
4. 研究查询仅在 ✅ 页面填写，提供 2-4 个精准搜索查询
5. **数据需求**仅在 ✅ 页面填写，描述该页需要的具体数据类型和维度；❌ 页面填"-"。数据需求必须具体化，如"2023-2026年全球AI市场规模及CAGR"、"中美欧AI监管政策对比"、"Top5 AI公司市场份额排名"，禁止模糊表述如"市场数据"、"相关统计"
6. **标题必须是结论性完整句**（Action + Result），如"AI Agent 市场规模突破千亿美元"
7. key_points 具体有信息量，优先使用调研中已发现的事实
8. 只输出 Markdown 格式大纲

### 大纲格式

```markdown
# 大纲：{topic}

**受众**：{audience}
**总页数**：{total_pages}
**叙事主线**：{narrative_arc}

## 页面大纲

| 页码 | 类型 | 标题 | 研究需求 | 研究查询 | 数据需求 | 视觉策略 |
|------|------|------|:--------:|----------|----------|----------|
| 1 | intro | 封面 | ❌ | - | - | 简洁大气 |
| 2 | trend | 市场规模突破千亿 | ✅ | "AI agent market 2026", "智能体市场规模预测" | 全球/中国AI Agent市场规模(2023-2026)、CAGR、Top5厂商份额 | 数据卡片+趋势图 |

## 详细要点

### P1: 封面
- **类型**：intro
- **标题**：{title}
- **内容概要**：{content_plan}

### P2: {title}
- **类型**：{type}
- **研究查询**：{research_queries}
- **数据需求**：{data_needs}
- **要点**：{key_points}
```

## Stage 3: 广度检索

### 3.1 研究页面识别

从 `outline.md` 筛选研究需求为 ✅ 的页面，提取其**研究查询**和**数据需求**。不需要研究的类型：封面/目录/总结/展望/方法框架。

### 3.2 查询优化

1. 中文主题用中英双语查询
2. 添加时间限定词（"2026"、"latest"）
3. 添加可信来源关键词（"report"、"白皮书"）
4. 避免过于宽泛的查询

### 3.3 执行搜索

**所有搜索在单条消息中并行发出**，禁止逐个串行。每页搜索由三部分组成：

1. **研究查询搜索**（来自页面大纲的「研究查询」列）：每页 2-4 次搜索
2. **数据需求定向搜索**（来自页面大纲的「数据需求」列）：根据数据需求中的每个具体数据维度，生成 1-2 个精准搜索查询。搜索模板：
   - 量值型数据：`"{数据需求关键词} 统计数据/报告"`、`"{data_needs_keyword} statistics report"`
   - 排名型数据：`"{数据需求关键词} 排名/Top N"`、`"{data_needs_keyword} ranking market share"`
   - 对比型数据：`"{数据需求关键词} 对比/比较"`、`"{data_needs_keyword} comparison benchmark"`
   - 趋势型数据：`"{数据需求关键词} 历年趋势/增长"`、`"{data_needs_keyword} historical trend growth"`
3. **页面类型兜底搜索**：当数据需求定向搜索未覆盖时序/对比需求时，按页面类型补充：
   - trend/data/comparison/technology → 时序搜索：`"{主题关键词} 市场规模 历年数据/趋势"`、`"{主题关键词} market size historical data"`
   - comparison/data/technology → 对比搜索：`"{主题关键词} 对比 {主要竞品/区域/技术路线}"`、`"{主题关键词} comparison vs competitors/benchmark"`、`"{主题关键词} market share ranking"`

每页合计 4-8 次搜索，总计 25-50 次。

### 3.4 常见主题搜索策略

**AI/技术类**：趋势预测 + 市场报告 + 企业案例 + 技术架构 + 政策支持 + Gartner hype cycle
**商业/行业类**：行业报告 + 竞争格局 + 头部企业对比 + 投融资动态 + 商业模式创新
**政策/社会类**：政策法规 + 国家战略 + 社会影响研究 + 国际比较

### 3.5 广度检索输出

每页收集：候选 URL（5-15 个）、来源简要摘要（1-2 句）、可信度初评（A+/A/A-/B+/B/C）、**数据需求覆盖度**（每条数据需求是否找到至少 1 个来源，未覆盖的标记为缺口）。

**URL 规范**：每条 URL 必须指向具体文章/报告页面（如 `https://www.news.cn/tech/20260414/2ac0cb963ff44e049f44b24069a6cdc0/c.html`），禁止仅记录首页域名（如 `https://www.news.cn/`）。搜索结果返回的每条结果均含完整链接，必须原样记录，不得截断为域名。

来源可信度评分标准：
- **A+ (90-100)**：权威机构（政府、国际组织）
- **A (80-89)**：企业官方（年报、财报）
- **A- (70-79)**：学术论文
- **B+ (65-69)**：权威媒体
- **B (60-64)**：行业媒体
- **C (<60)**：自媒体/内容农场（**排除**）

排除条件：纯观点无数据、来源不明的二手转述、商业推广、可信度 <60。

## Stage 4: 反思与迭代

### 4.1 覆盖度评估

逐页对照：每页是否 3+ 条独立证据？高可信来源(B+及以上)占比是否达标？核心论点是否缺乏支撑？**数据需求是否全部覆盖**（每条数据需求至少有 1 个来源支撑，未覆盖的即为缺口）？

### 4.2 缺口分析

- 核心问题未找到答案
- 关键数据仅单一来源
- 来源类型偏斜（如全是媒体，无权威报告）
- 数据需求缺口（由 4.1 覆盖度评估识别）
- 大纲未预见的重要发现（需新增页面或调整结构）

### 4.3 迭代执行

| 缺口页面数 | 决策 |
|-----------|------|
| 0-1 页 | 补充 1 轮搜索后进入 Stage 5 |
| 2-3 页 | 补充 2 轮搜索 |
| 3+ 页 | 补充 2-3 轮后重新评估 |

迭代终止：达到深度最大轮数（L1:1轮, L2:2轮, L3:3轮），或连续一轮无重大缺口。补充搜索仍须并行发出。如需调整大纲，修订 `outline.md`。

## Stage 5: 深度抓取

### 5.1 来源筛选

每页选取 **3-5 个高可信来源**（B+ 及以上）进行深度抓取。优先级：权威机构报告 > 白皮书/论文 > 知名媒体深度报道 > 官方数据 > 企业官方。排除内容农场、无来源转述、>2年旧信息（经典案例除外）。

### 5.2 抓取执行

使用 WebFetch 抓取正文，并行抓取同一批次 URL。提取：
1. **关键事实**：具体数据点、统计数字
2. **核心观点**：报告/文章的核心论断
3. **案例信息**：具体公司、产品、实施情况
4. **引用来源**：原始数据来源和发布时间
5. **时序数据**：同一指标在不同年份/季度的数值（至少 3 个时间点），用于趋势图生成；优先提取有连续年份的序列数据
6. **对比数据**：同一指标在不同对象/类别间的数值对比（至少 2 个对比对象、2 个对比维度），用于条形图/雷达图/对比卡片生成；关注竞品对比、区域对比、技术路线对比等

### 5.3 幽灵来源识别

标记高风险特征：无URL或DOI不匹配、标题/年份与内容矛盾、无法回溯的二手转述、引用来源与领域不符、发布时间异常。尝试补充验证，无法核实则排除。

## Stage 6: 成果撰写

### 6.1 生成 research.md

将深度抓取结果按 [研究报告模板](./templates/research_output_template.md) 格式写入 `{output_dir}/research.md`。

### 6.2 写作硬规则（11 条）

1. **叙事优先**：≥80% 正文为连贯散文段落，禁止用 bullet-point 替代论证主体
2. **精准引用**：每个事实陈述（数据/比例/日期/因果）首次出现时同句内附 [N] 引用；禁止"研究表明""据统计""专家认为"等伪引用
3. **反空泛**：禁止"市场前景广阔""发展迅速""越来越多"等无来源修饰；禁止"某知名企业""部分公司"等模糊案例；用精确数字替代形容词
4. **数据完整保留**：所有 key_findings 中的 data_point 必须完整呈现在"关键数据清单"表格中，每条数据点独占一行，包含数据项、数值/结果、来源引用和时间；禁止使用斜杠分隔的扁平列表
5. **来源可溯源**：来源汇总表和每页引用来源中，URL 必须填写具体文章/报告的完整网址（如 `https://www.news.cn/tech/20260414/.../c.html`），禁止仅填写首页域名（如 `https://www.news.cn/`），禁止用"-"或留空，确保每条证据可直接访问验证
6. **深度达标**：每页研究分析 ≥3 个实质段落、≥400 字；全文达到深度字数要求（L1≥3k/L2≥5k/L3≥8k）
7. **表格规范**：关键数据清单必须使用 Markdown 表格格式（数据项 | 数值/结果 | 来源 | 时间），禁止使用斜杠分隔的扁平列表；对比数据表格格式为（对比维度 | 对象A | 对象B | ... | 来源），支持 2-N 个对比对象
8. **素材充实**：核心论点必须编号展开（每条附带一句说明）；案例素材必须包含实体名称、简要描述和来源引用；建议可视化方案需指定图表类型及理由
9. **数据清单充实**：每页关键数据清单必须 ≥5 条数据点，且至少覆盖 2 种数据类型（绝对值/百分比/排名/增长率）；数据点应尽量包含可构成时间序列的多年数据（如"2023年X亿，2024年Y亿，2025年Z亿"），为下游图表生成提供充足素材
10. **时序数据优先**：类型为 trend/data/comparison/technology 的页面必须提供至少 1 组时序数据（≥3 个时间点），用于生成趋势图；如原始来源缺乏时序数据，应在 Stage 3 广度检索中主动搜索"XXX 市场规模 历年数据/趋势"来补全
11. **对比数据强制**：类型为 comparison/data/technology 的页面必须提供至少 1 组对比数据（≥2 个对比对象、≥2 个对比维度），用于生成条形图/雷达图/对比卡片；如原始来源缺乏对比数据，应在 Stage 3 广度检索中主动搜索"XXX 对比 竞品/benchmark"来补全；其他类型页面如有明确对比维度也应提供

## Stage 7: 验证与交付

### 7.1 交叉验证

- L3：关键数据 3+ 独立来源确认，幽灵来源检查必执行
- L2：关键数据 2+ 独立来源确认，单源数据标注"（单源）"
- L1：标注所有单源数据

矛盾数据：标注不同来源，分析矛盾原因（口径/时间/定义差异），呈现多方数据而非强行统一。

### 7.2 一致性检查

- 引用编号连续无跳跃重复
- 引用与来源汇总表一致
- 可信度评分与六档体系一致

### 7.3 产物验证

**outline.md**：文件存在非空，含 `# 大纲：` 标题和 `## 页面大纲` 章节，每个 ✅ 页面有研究查询和数据需求。

**research.md**：文件存在非空，含 `## 研究概述`、`## 大纲总览`、`## 逐页研究成果`，每个研究页面有 `### P{N}:` 小节含 `#### 研究分析` 和 `#### PPT 内容建议`，PPT 内容建议中包含关键数据清单表格。

### 7.4 质量核心检查（10 项）

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 证据密度 | 每页 ≥3 条 key_findings |
| 2 | 来源数量 | 总来源 ≥ 研究页面数 × 2 |
| 3 | 数据支撑 | ≥50% findings 含具体数据点，且每页关键数据清单 ≥5 条 |
| 4 | 引用规范 | 每页 ≥3 个 [N] 引用，无伪引用 |
| 5 | 反空泛 | 无模糊修饰/模糊案例/占位文本 |
| 6 | 字数达标 | L1≥3k/L2≥5k/L3≥8k |
| 7 | 数据清单表格化 | 每页关键数据清单为 Markdown 表格，非斜杠分隔扁平列表 |
| 8 | 素材充实度 | 核心论点有展开说明；案例含实体名称；有可视化方案建议 |
| 9 | 时序数据覆盖 | 研究页面中 trend/data/comparison/technology 类型必须有 ≥1 组时序数据（≥3 个时间点） |
| 10 | 对比数据覆盖 | 研究页面中 comparison/data/technology 类型必须有 ≥1 组对比数据（≥2 个对比对象、≥2 个对比维度） |

### 7.5 错误处理

| 问题 | 处理 |
|------|------|
| 搜索无结果 | 扩大查询范围，尝试同义词或英文 |
| 来源不足（<3/页） | 标记"来源有限"，降级为基础描述 |
| 抓取失败 | 跳过 URL，用搜索摘要替代 |
| 信息矛盾 | 标注矛盾来源，提供多方观点 |
| 全部失败 | 报告失败原因，输出无研究数据的纯大纲 |
