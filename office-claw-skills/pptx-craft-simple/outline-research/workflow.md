## 工作流程概览

```
用户需求
  → Stage 0: 初始化输出目录（时间戳隔离）
  → Stage 1: 需求调研（Topic Investigation）
  → Stage 2: 大纲生成（基于调研结果）
  → Stage 3: 研究需求分析
  → Stage 4: 广度检索（Broad Search）
  → Stage 5: 反思与迭代
  → Stage 6: 来源核验
  → Stage 7: 深度抓取（Deep Crawl）
  → Stage 8: 研究成果结构化
  → Stage 9: 交叉验证与报告核实
  → Stage 10: 输出与验证
```

## Stage 0: 初始化输出目录

在执行任何研究工作之前，先确定输出目录。

### 0.1 判断路径模式

| 条件 | 模式 | 行为 |
|------|------|------|
| 用户未指定路径 | **自动生成** | 调用脚本在 `output_base` 下创建时间戳子目录 |
| 用户明确指定完整路径 | **用户指定** | 直接使用用户路径，必要时创建目录 |

**判断用户指定路径的标志**：
- 用户在需求中提及"输出到 X 目录"、"保存到 X 路径"
- 用户提供了完整的输出路径

**自动生成的标志**：
- 用户未提及任何路径相关要求
- 用户仅表示"默认即可"

### 0.2 自动生成时间戳目录（默认模式）

```bash
node {skill_root}/pptx-craft-simple/scripts/utils/generate_timestamp_dir.js [output_base]
```

- `output_base` 指定输出基础目录，如不传则默认在当前工作目录的 `output/` 下创建
- 传参数时，在指定的 `{output_base}` 目录下创建
- 脚本自动获取当前时间，格式化为 `YYYYMMDD_HHMMSS`
- 检查是否存在同前缀目录，自动递增序号（`_000`, `_001`, ...）
- 创建目录并返回完整路径

将脚本返回的路径赋值给 **`{output_dir}`** 变量，后续所有阶段使用此路径。

### 0.3 用户指定模式

直接将用户路径赋值给 `{output_dir}`，如目录不存在则创建：

```bash
mkdir -p {output_dir}
```

## Stage 1: 需求调研（Topic Investigation）

**目标**：在生成大纲之前，先对用户主题做快速摸底调研，了解领域全貌、关键维度和最新动态，为大纲生成提供事实基础。

### 1.1 需求标准化

收到用户需求后，先做结构化拆解：

- **主题提取**：从用户输入中识别核心主题
- **受众推断**：根据主题和上下文推断目标受众
- **页数确认**：使用用户指定页数或默认 10 页
- **风格目标**：专业、简洁、信息密度适中

### 1.2 快速调研执行

围绕用户主题发起 **5-8 次并行搜索**，目标是快速掌握：

1. **领域现状**：这个主题当前的整体格局是什么？
2. **关键维度**：业内讨论这个主题时，通常涉及哪些核心方面？
3. **最新动态**：近期有哪些重要事件、数据或趋势？
4. **核心玩家**：有哪些关键机构、公司或人物？
5. **争议与热点**：是否存在分歧观点或热门话题？

**搜索查询构造**：

```
[并行发出 3-5 次搜索]
WebSearch #1: "{topic} 综述 概览 {year}"          ← 领域全貌
WebSearch #2: "{topic} latest trends {year}"      ← 最新趋势（英文源）
WebSearch #3: "{topic} 关键技术/核心问题/主要挑战" ← 关键维度
WebSearch #4: "{topic} market report statistics"  ← 数据摸底
WebSearch #5: "{topic} 典型案例 企业实践"          ← 案例摸底（如适用）
```

**调研搜索量**：默认执行 5-8 次并行搜索，覆盖多语言、多角度，确保充分摸底。

### 1.3 调研结果整理

将搜索结果整理为结构化的调研摘要，包含：

```markdown
## 主题调研摘要：{topic}

### 领域概况
{2-3 句话概括该领域的整体现状}

### 关键维度
- {维度1}：{一句话说明}
- {维度2}：{一句话说明}
- {维度3}：{一句话说明}
- ...

### 重要发现
1. {发现1 — 附来源}
2. {发现2 — 附来源}
3. {发现3 — 附来源}

### 关键数据点
- {数据1 — 附来源}
- {数据2 — 附来源}

### 核心玩家/案例线索
- {玩家/案例1}
- {玩家/案例2}

### 大纲建议方向
基于以上调研，建议大纲覆盖以下方向：
1. {建议方向1}
2. {建议方向2}
3. ...
```

### 1.4 调研输出

将调研摘要存入执行上下文，**不写入文件**。

**此内容直接作为 Stage 2 大纲生成的输入依据。**

## Stage 2: 大纲生成（基于调研结果）

**目标**：结合用户需求和需求调研结果，生成有事实支撑的结构化大纲。

### 2.1 大纲生成

使用 Stage 1 存入上下文中的调研内容，直接注入大纲生成 Prompt：

```
# Role
你是一位顶级的 PPT 结构架构师，精通金字塔原理。

# Context
- 主题：{topic}
- 目标受众：{audience}
- 页数要求：{page_count} 页

# 背景调研信息（Context）
以下是关于该主题的搜索调研摘要，请务必参考这些信息来规划大纲，
使其切合当前的市场现状和技术事实，而不是凭空捏造。

{topic_investigation_content}

# Core Methodology: 金字塔原理
1. 结论先行：每个部分以核心观点开篇
2. 以上统下：上层观点是下层内容的总结
3. 归类分组：同一层级的内容属于同一逻辑范畴
4. 逻辑递进：内容按照某种逻辑顺序展开

# Task
基于以上调研信息，生成一份完整的 PPT 大纲，严格按以下 Markdown 格式输出：

# 大纲：{topic}

**受众**：{audience}
**总页数**：{total_pages}
**叙事主线**：{narrative_arc}

## 页面大纲

| 页码 | 类型 | 标题 | 研究需求 | 研究查询 | 视觉策略 |
|------|------|------|:--------:|----------|----------|
| 1 | intro | 封面 | ❌ | - | 简洁大气 |
| 2 | trend | 市场规模突破千亿 | ✅ | "AI agent market 2026", "智能体市场规模预测" | 数据卡片 |
...（根据实际页面填写）

## 详细要点

### P1: 封面
- **类型**：intro
- **标题**：{title}
- **内容概要**：{content_plan}
- **要点**：{key_points}

### P2: {title}
- **类型**：{type}
- **标题**：{title}
- **内容概要**：{content_plan}
- **要点**：{key_points}
- **研究查询**：{research_queries}
- **数据需求**：{data_needs}

# Rules
1. 大纲必须基于调研信息中的真实发现，不得编造不存在的趋势或数据
2. type 为 intro/agenda/conclusion 的页面研究需求标记为 ❌
3. type 为 trend/data/case/comparison/technology 的页面研究需求标记为 ✅
4. 研究查询仅在需要研究的页面填写，提供 2-4 个精准搜索查询
5. title 必须是结论性完整句（Action + Result），可引用调研中的发现来拟定
6. key_points 应具体、有信息量，优先使用调研中已发现的事实
7. 如果调研显示某方向信息不足或已过时，不要将其作为核心页面
8. 只输出 Markdown 格式的大纲，不包含其他文字
```

### 2.2 大纲输出

将生成的大纲写入 `{output_dir}/outline.md`。

## Stage 3: 研究需求分析

### 3.1 识别研究页面

遍历 `outline.md` 中的页面列表，筛选"研究需求"为 ✅ 的页面。

### 3.2 研究需求判定规则

| 内容类型 | need_research | 原因 |
|----------|:---:|------|
| 趋势判断 | ✅ | 依赖时效信息 |
| 市场数据 | ✅ | 需要具体数字支撑 |
| 技术突破 | ✅ | 需要最新进展 |
| 案例对比 | ✅ | 需要真实案例 |
| 政策动态 | ✅ | 需要权威来源 |
| 封面/目录 | ❌ | 无需外部信息 |
| 总结/展望 | ❌ | 基于前文综合 |
| 方法框架 | ❌ | 通用知识即可 |

### 3.3 搜索查询优化

对每个需要研究的页面，从大纲 Markdown 中提取"研究查询"列的内容：

1. 确保查询语言与目标来源匹配（中文主题用中英双语查询）
2. 添加时间限定词（如 "2025"、"2026"、"latest"）
3. 添加可信来源关键词（如 "report"、"statistics"、"白皮书"）
4. 避免过于宽泛的查询

---

## Stage 4: 广度检索（Broad Search）

### 4.1 目标

铺开信息面，获取高质量候选来源。

### 4.2 执行策略

**并行执行**：将所有需要研究的页面的搜索查询汇总，在单条消息中并行发出所有搜索请求。

```
[单条消息并行发出所有搜索]
WebSearch #1: "{page2_query1}"
WebSearch #2: "{page2_query2}"
WebSearch #3: "{page3_query1}"
WebSearch #4: "{page3_query2}"
WebSearch #5: "{page4_query1}"
...
```

**❌ 错误（逐个串行）**：
```
WebSearch #1 → 等结果 → WebSearch #2 → 等结果 → ...
```

**✅ 正确（全部并行）**：
```
所有搜索在一条消息中同时发出
```

### 4.3 搜索量

每个研究页面执行 4-6 次搜索，总搜索预算 20-40 次，确保信息覆盖充分。

### 4.4 广度检索输出

为每个研究页面收集：
- 候选 URL 列表（5-15 个）
- 每个来源的简要摘要（1-2 句）
- 来源可信度初步评分（高/中/低）

---

## Stage 5: 反思与迭代（Reflect & Iterate）

### 5.1 覆盖度评估

将 Stage 4 收集的信息与大纲逐页对照：
- 每页是否获得 3+ 条独立证据？
- 高可信来源（65+分）占比是否达标？
- 核心论点是否缺乏支撑？

### 5.2 信息缺口分析

识别以下类型的信息缺口：
- **核心问题未找到答案**：用户关注的核心问题在搜索结果中完全缺失
- **数据点仅单一来源**：关键数据只有一个来源支持
- **信息来源类型偏向**：过度依赖某一类型来源（如全是媒体，无权威报告）

### 5.3 新维度识别

识别大纲未预见的重要发现：
- 原有页面划分不合理（某主题信息过载，另一主题几乎无内容）
- 需新增页面的重要趋势（新发现的重要方向值得独立成页）
- 原有假设被推翻（初步假设与新证据矛盾）

### 5.4 动态大纲调整

根据以上三维评估，动态调整：
1. **修订 outline.md**：如发现新维度或原有假设被推翻，更新页面结构
2. **生成补充查询**：针对缺口生成精准补充搜索
3. **迭代终止**：已达深度要求且无重大缺口时，终止迭代

### 5.5 迭代决策

| 缺口页面数 | 决策 |
|-----------|------|
| 0-1 页 | 补充 1 轮搜索后进入 Stage 7 |
| 2-3 页 | 补充 2 轮搜索 |
| 3+ 页 | 补充 2-3 轮后重新评估 |

### 5.6 迭代终止条件

- 达到当前深度的最大迭代轮数（L1: 1 轮, L2: 2 轮, L3: 3 轮）
- 连续一轮反思未发现重大信息缺口

**迭代执行方式**：
- 根据缺口分析生成补充查询
- 在单条消息中并行发出所有补充搜索
- 补充搜索完成后，返回 5.1 重新评估覆盖度

## Stage 6: 来源核验（Source Verification）

### 6.1 来源可信度评分

对所有候选来源按六档体系评分：

| 分值 | 等级 | 来源类型 |
|:----:|:----:|----------|
| 90-100 | A+ | 权威机构（政府、国际组织） |
| 80-89 | A | 企业官方（年报、财报） |
| 70-79 | A- | 学术论文 |
| 65-69 | B+ | 权威媒体 |
| 60-64 | B | 行业媒体 |
| <60 | C | 自媒体/内容农场（排除） |

### 6.2 幽灵来源识别

识别以下高风险特征并标记：
- 无 URL、DOI 与出版信息不匹配
- 标题/年份与内容矛盾
- 无法回溯的二手转述
- 引用来源与内容领域不符
- 发布时间异常

**处理流程**：
1. **标记**：识别幽灵来源特征
2. **补充**：尝试补充可验证信息
3. **降级/移除**：无法核实则降级或移除

### 6.3 来源筛选

| 要求层级 | 筛选标准 |
|---------|----------|
| 必选 | 可信度 ≥ 60 |
| 优先 | 可信度 ≥ 65 |
| 排除 | 可信度 < 60 |

---

## Stage 7: 深度抓取（Deep Crawl）

### 7.1 目标

从候选来源中提取可上页的结构化信息。

### 7.2 来源筛选

从广度检索结果中，为每个研究页面选取 **3-5 个高可信来源**进行深度抓取：

**来源优先级**：
1. 权威机构报告（McKinsey、Gartner、IDC 等）
2. 行业白皮书和研究论文
3. 知名媒体深度报道（MIT Tech Review、Bloomberg 等）
4. 官方数据统计（政府、行业协会）
5. 头部企业官方发布

**排除来源**：
- 内容农场和低质量聚合站
- 无明确来源的二手转述
- 过于陈旧的信息（>2年，除非是经典案例）

### 7.3 深度抓取执行

对筛选出的高质量 URL，使用 `WebFetch` 工具抓取正文内容，提取：

1. **关键事实**：具体数据点、统计数字
2. **核心观点**：报告/文章的核心论断
3. **案例信息**：具体公司、产品、实施情况
4. **引用来源**：原始数据来源和发布时间

**并行抓取**：同一批次的多个 URL 在单条消息中并行抓取。

### 7.4 信息提炼

每个页面的抓取结果提炼为以下结构：

```json
{

  "page_title": "页面标题",
  "research_body": {

      "analysis_paragraphs": [
```

## Stage 8: 研究成果结构化

### 8.1 研究数据汇总

将所有页面的研究数据存入执行上下文，**不写入文件**。

此内容直接作为 Stage 8.2 的输入。

### 8.2 生成 research.md

使用 Stage 8.1 存入上下文中的研究数据，直接生成可读的 Markdown 报告，格式参考 [研究报告模板](./templates/research_output_template.md)。

#### 写作标准（强制执行）

**叙事优先**：每页研究分析以连贯的散文段落为主体，≥80% 正文内容为流畅段落，禁止用 bullet-point 列表替代论证主体。

**数据完整保留原则**：所有 `key_findings` 中包含 `data_point` 的条目，**必须**在 research.md 中完整呈现，不得将多条独立数据点合并为一句模糊描述。每条数据应同时保留：具体数值 + 来源编号 + 发布日期（如有）。呈现位置可选：研究分析段落（叙述形式）或"关键数据清单"表格（结构化形式）。对含 5+ 条数据点的页面，**允许**超过 20% 以结构化列表/表格呈现。

**精准引用**：
- 每个事实陈述（数据、比例、日期、因果判断）在首次出现时，必须在同一句内附 [N] 引用标注
- ✅ "根据 Gartner 2026 年报告，全球 AI Agent 市场规模预计达到 1200 亿美元 [3]"
- ❌ "市场规模持续扩大"（无来源、无数据）

**反空泛规则**：
- 禁止无来源修饰："市场前景广阔"、"发展迅速"、"越来越多的企业"
- 禁止伪引用："研究表明"、"据统计"、"专家认为"（必须指明具体来源）
- 禁止模糊案例："某知名企业"、"部分公司"（必须用具体公司名）
- 用精确数字替代形容词："渗透率从 12% 提升至 37% [2]"而非"渗透率显著提升"

**段落深度**：
- 每页研究分析 ≥3 个实质段落（不含标题和引用来源列表）
- 每页研究分析总字数 ≥400 字（不含 PPT 内容建议和引用来源部分）
- 每段有完整结构：论点 → 证据 → 推论

**展示要点数量**：
- 每页 `talking_points` 建议 6-10 个，高密度页可达 10+ 个
- 研究内容丰富时，优先保留更多要点而非强制压缩

**事实与分析分离**：
- 引用来源的事实："根据 [N]……"、"[N] 报告……"
- 自己的分析推断："这表明……"、"综合来看……"
- 不确定的内容："尚未有公开数据……"而非编造引用

**写作节奏**：
```
事实/数据+引用 → 分析/推论（无引用） → 新事实/数据+引用 → 总结（无引用）
```
- 禁止同一来源的重复连续引用（如 [3][3][3]）
- 每段约 1-3 个引用标记
- 引用分散在不同句子中，避免连续 3 句以上都有引用

---

## Stage 9: 交叉验证与报告核实（Cross-Validation & Report Verification）

### 9.1 关键数据点最终验证

对 research.md 中的关键数据点进行最终核实：
- 具体数字（市场规模、增长率、渗透率）
- 预测性结论（未来趋势、市场份额预测）
- 因果关系陈述（"因 X 导致 Y"）

### 9.2 深度特定验证要求

| 研究深度 | 验证要求 |
|---------|---------|
| L3 | 关键数据 3+ 独立来源确认；幽灵来源检查必须执行；无单源关键数据 |
| L2 | 关键数据 2+ 独立来源确认；单源数据需标注"（单源）" |
| L1 | 标注所有单源数据；不强制要求多源验证 |

### 9.3 矛盾数据处理核实

检查 research.md 中矛盾数据是否已正确处理：
- 矛盾数据是否标注了不同来源
- 是否分析了矛盾原因（口径/时间/定义差异）
- 是否呈现了多方数据而非强行统一

### 9.4 报告一致性检查

- 引用编号连续性（无跳跃、无重复）
- 引用与来源汇总表一致
- 可信度评分与六档体系一致

## Stage 10: 输出与验证

### 10.1 产物验证清单

执行以下检查：

1. **outline.md 验证**：
   - 文件存在且非空
   - 包含 `# 大纲：` 标题
   - 包含 `## 页面大纲` 章节
   - 包含页面列表表格（至少 total_pages 行）
   - 每个需要研究的页面有研究查询

2. **research.md 验证**：
   - 文件存在且非空
   - 包含 `## 研究概述` 章节
   - 包含 `## 大纲总览` 章节
   - 包含 `## 逐页研究成果` 章节
   - 每个研究页面有对应的 `### P{N}:` 小节，且包含 `#### 研究分析` 子标题
   - 每个研究页面有对应的 `#### PPT 内容建议` 子标题

### 10.2 质量检查（21 项）

#### 结构完整性（4 项）

1. **outline.md 标题**：文件包含 `# 大纲：` 标题
2. **## 页面大纲**：文件包含 `## 页面大纲` 章节
3. **research.md ## 研究概述**：文件包含 `## 研究概述` 章节
4. **## 逐页研究成果**：文件包含 `## 逐页研究成果` 章节

#### 内容质量（6 项）

5. **证据密度**：每个研究页面至少 3 条 `key_findings`
6. **来源多样性**：总来源数 ≥ 研究页面数 × 2
7. **数据支撑**：至少 50% 的 findings 包含具体数据点
8. **来源可信度**：至少 30% 的来源可信度达 65+（B+ 及以上）
9. **段落深度**：每页"研究分析"部分 ≥3 个实质段落，总字数 ≥400 字
10. **字数要求**：全文达到研究深度对应的最低字数（L1≥3,000/L2≥5,000/L3≥8,000 字）

#### 引用规范（4 项）

11. **引用密度**：每页"研究分析"部分至少出现 3 个 [N] 内联引用
12. **无"同上"引用**：不出现"同上"等模糊引用指示
13. **引用格式正确**：所有 [N] 引用在来源汇总表中可查
14. **禁止伪引用**：无"研究表明""据统计""专家认为"等无来源修饰

#### 反空泛（3 项）

15. **无模糊修饰**：不出现"市场前景广阔""发展迅速""越来越多"等无来源修饰
16. **无模糊案例**：不出现"某知名企业""部分公司"等模糊案例描述
17. **无占位文本**：不出现"待补充""TODO""待验证"等占位文本

#### 可信度验证（2 项）

18. **幽灵来源检查**：所有来源有 URL/可追溯元数据，无幽灵来源特征
19. **核心论点验证**：L3 研究中，关键数据点有 3+ 独立来源确认

#### 数据完整性（2 项）

20. **数据清单完整**：每个研究页面的"关键数据清单"包含该页 **100%** 的 `key_findings` 数据点（不允许遗漏）
21. **数据溯源**：所有数据点包含 publish_date（如有），且引用编号 [N] 可在引用来源中查找到对应条目

### 10.3 错误处理

| 问题 | 处理方式 |
|------|----------|
| 搜索无结果 | 扩大查询范围，尝试同义词或英文查询 |
| 来源不足（<3 个/页） | 标记为"来源有限"，降级为基础描述 |
| 抓取失败 | 跳过该 URL，使用搜索摘要替代 |
| 信息矛盾 | 标注矛盾来源，提供多方观点 |
| 全部失败 | 报告失败原因，输出无研究数据的纯大纲 |
